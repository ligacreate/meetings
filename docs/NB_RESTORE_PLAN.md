# NB-RESTORE — план восстановления админки meetings (notebooks/questions/cities)

**Дата:** 2026-05-05
**Severity:** **P1**
**Срочность:** **после FEAT-002 этап 4.** Не блокер этапа 4.
**Источник:** SEC-PINS Variant A (полный выпил) + verification 2026-05-05 (POST /notebooks без JWT → 503/401 + PostgREST 42501).
**Связанные документы:** [SEC_PINS_2026-05-05.md](SEC_PINS_2026-05-05.md), [ARCH_CHECK_meetings_admin_2026-05-05.md](ARCH_CHECK_meetings_admin_2026-05-05.md), [GARDEN_QA_2026-05-05.md](GARDEN_QA_2026-05-05.md)

---

## Что произошло и что нужно

После phase 18 миграции в Garden (2026-05-04) роль `web_anon` имеет только `GRANT SELECT` на `events`/`cities`/`notebooks`/`questions`. Анонимные write-вызовы из meetings/admin падают 401. Внутри Sec-PINS таска админка meetings выпиливается полностью (см. [SEC_PINS](SEC_PINS_2026-05-05.md)).

**Что Ольга реально использовала:**
- **Notebooks** — загрузка контентных тетрадей издательства (title, description, image_url base64, pdf_url). Уникально, не дублируется Garden.
- **Questions** — добавление/удаление вопросов рефлексии (одиночное и bulk). Уникально.
- **Cities** — справочник городов (POST/DELETE по name). Уникально, неявно использовался при создании события с новым городом.

После полного выпила Ольга **не сможет** загрузить новую тетрадь, вопрос или город через UI. Старые данные продолжают читаться через `web_anon SELECT` — публичная часть сайта работает.

**Цель этого таска:** вернуть Ольге возможность редактировать тетради/вопросы/города с **нормальной защитой**, без хардкоженных PIN-кодов.

---

## Три опции — для обсуждения стратегом

### Вариант 1 — Telegram-Auth → JWT → PostgREST `role=admin` (как Garden)

**Архитектурно:** ближе всего к существующей Garden-архитектуре. Garden уже использует Telegram-Auth (`auth.skrebeyko.ru` + JWT в `localStorage('garden_auth_token')`, см. Garden's `services/dataService.js:7-44`). Тот же механизм можно встроить в meetings.

**Что нужно:**
- В meetings подключить логин через Telegram (PIN-код заменяется на «Войти через Telegram»).
- После логина клиент кладёт JWT в localStorage и передаёт `Authorization: Bearer ...` во все write-запросы.
- В Garden / PostgREST настроить, чтобы JWT с `role=admin` имел `INSERT/UPDATE/DELETE` на `notebooks`/`questions`/`cities` (плюс при необходимости `events` — но это spike).
- Возможно, нужна синхронизация ролей между Garden и meetings (Ольга — `admin` и там, и там; на будущее: какой email/Telegram-id даёт права в meetings? только Ольгин или ещё кому-то?).

**Плюсы:**
- Единая модель аутентификации с Garden — одна страница логина на двух доменах, общий пользователь (если запросы идут на тот же `auth.skrebeyko.ru`).
- Защита уровня production-grade (JWT с подписью, expiration, refresh).
- Чистый код фронта: `Authorization` уходит вместе с любым write, ничего хардкодить.
- Аудит-лог в БД может видеть, кто именно изменил запись (через JWT claims).

**Минусы:**
- **Самый большой scope из трёх.** Нужны:
  - изменения в Garden auth-сервисе (whitelist для meetings-домена в Allowed Origins / refresh-token флоу);
  - GRANT'ы в БД для роли `admin` на 3-4 таблицы;
  - integration-тест на full флоу логина + write;
  - Разработка login-UX в meetings (форма Telegram-Auth с QR/redirect).
- Зависит от Garden-команды (не самостоятельный таск meetings).
- Если Garden-Auth недоступен (SLA `auth.skrebeyko.ru`) — админка meetings ложится.

**Время:** оценочно 2-3 дня (1 день Garden-side, 1 день meetings-side, 0.5 дня тесты + smoke).

**Риск регрессии:** низкий для публичной части (JWT добавляется только в admin-флоу, GET остаётся анонимным). Высокий для админ-флоу при ошибках в Garden-side.

**Влияние на FTP-deploy `dangerous-clean-slate: true`:** ноль, никаких серверных конфигов meetings не трогает.

### Вариант 2 — Caddy/Apache Basic Auth + serv-side JWT injection

**Архитектурно:** простейшая защита периметра. На уровне Apache (`.htaccess`) или Caddy на `meetings.skrebeyko.ru/admin/*` ставится HTTP Basic Auth (логин-пароль из ENV). Сервер инжектит сервисный JWT в каждый PostgREST-запрос (через прокси-эндпоинт), либо meetings-фронт получает короткоживущий JWT после Basic Auth и кладёт в localStorage.

**Что нужно:**
- `mod_auth_basic` на Apache + AuthUserFile (хеш-пароли) с креденшелами вне репо, либо Caddy с `basicauth` директивой.
- Сервисный JWT с `role=admin` — выдан Garden-команде, прописан как ENV-переменная на сервере.
- На фронте: убрать клиентскую PIN-проверку; `/admin` за Basic Auth; после логина положить JWT в localStorage (или прокси-эндпоинт `/api/admin/*` инжектит JWT серверно).

**Плюсы:**
- Самый простой по коду фронта (PIN был, теперь Basic Auth — одна строка в `.htaccess`).
- Не зависит от Telegram-Auth.
- Можно развернуть в течение часа, если есть SSH-доступ.
- Защита уровня «никто посторонний не зайдёт».

**Минусы:**
- **FTP-deploy `dangerous-clean-slate: true` ([deploy.yml:32](../.github/workflows/deploy.yml#L32)) затирает всё на сервере перед загрузкой.** Если `.htaccess` с Basic Auth и `AuthUserFile` лежат вне `dist/` — после первого деплоя они снесутся вместе со всем остальным. Альтернативы:
  - положить `.htaccess` **в `dist/`** (мержится с public/.htaccess через [public/.htaccess](../public/.htaccess) копирование при build) — но тогда `AuthUserFile` тоже нужен в репо, что либо хранит хеш пароля в git, либо нужна синхронизация при деплое.
  - **мигрировать с FTP-clean-slate на rsync через SSH** — отдельная работа, см. AUDIT P2-5.
  - Caddy вместо Apache (если перешли на Caddy) — конфиг живёт отдельно, FTP его не трогает. Это инфраструктурное изменение.
- Сервисный JWT — long-lived, попадёт в каждый запрос: если кто-то увидит в DevTools — получит admin-доступ к БД. Mitigation: ротация раз в N месяцев + scope JWT только на нужные таблицы.
- Basic Auth непопулярен у пользователей (модальное окно браузера, нет logout). UX хуже Telegram-Auth.

**Время:** оценочно 0.5-1 день при условии rsync-миграции; 2-3 часа без неё (но с риском, что следующий FTP-deploy всё снесёт).

**Риск регрессии:** средний. Основной риск — забыть про clean-slate и потерять конфиг при деплое.

### Вариант 3 — переезд в Garden, удаление meetings/admin навсегда

**Архитектурно:** лучший. У Garden уже есть AdminPanel с Telegram-Auth, JWT, role=admin, аудит. Добавить туда вкладки «Тетради» / «Вопросы» / «Города» — и всё.

**Что нужно:**
- В Garden's `views/AdminPanel.jsx` добавить 3 вкладки: Notebooks, Questions, Cities.
- В Garden's `services/dataService.js` добавить методы `getNotebooks/addNotebook/updateNotebook/deleteNotebook`, аналогично для questions и cities.
- БД: GRANT'ы для роли `admin` на 3 таблицы (как в Варианте 1, но там использует та же роль).
- meetings — никаких изменений: продолжает анонимный read через PostgREST.

**Плюсы:**
- **Архитектурно самое чистое.** Один админ-домен, одна точка входа, одна модель пользователей и ролей. Никакой дубликации между двумя репо.
- Garden-команда уже знает свой стек, любые правки идут по проторённому пути.
- meetings остаётся **полностью read-only** — после SEC-PINS apply там нет ни одного write-вызова, никакой админки. Зона ответственности супер-чёткая: «meetings = публичная читалка».
- Бесплатно решает все будущие проблемы: новые поля админки → в Garden; ротация прав → в Garden; аудит → в Garden.

**Минусы:**
- **Самый большой scope из трёх.** Нужны:
  - 3 новые вкладки UI в Garden (формы create/update/delete с превью + загрузка PDF).
  - Хендлинг загрузки PDF и base64 image — у Garden другой паттерн (storage.from('avatars')), нужно либо завести `notebooks-pdf` bucket в Supabase Storage, либо переехать на URL вместо base64 (см. AUDIT P2-3, P2-6 — раз уж переезжаем, разумно сразу нормально).
  - Garden-команда должна найти время на это в своём бэклоге.
- Завязка на занятость Garden-стратега / Claude Code в Garden.
- meetings-репо при этом **не закрывается** — он по-прежнему хостит публичный сайт, читает PostgREST. Только админка уезжает.

**Время:** оценочно 3-5 дней Garden-команды (UI + dataService + БД + smoke).

**Риск регрессии:** низкий для meetings (тут ничего не меняется). Средний для Garden (новые админ-формы, новые БД-grants).

**Влияние на FTP-deploy:** ноль.

## Рекомендация

**Вариант 3 (переезд в Garden) — стратегически.**
- Лучшая архитектура. Один источник правды. meetings = публичная читалка с zero write surface — это и описание сайта, и реальная защита, и легче будущая поддержка.
- Снимает класс проблем «дублирование между meetings и Garden» навсегда.
- Garden-команда уже умеет всё, что нужно (auth, JWT, GRANT'ы, формы). Сложность не выше, чем В1.

**Вариант 1 (Telegram-Auth в meetings) — компромисс**, если Garden-команда сейчас перегружена и стратег хочет распределить нагрузку. Но потом всё равно вылезет дубликат.

**Вариант 2 (Basic Auth) — только как stop-gap**, если Ольге срочно (за день) нужно загрузить тетрадь и нет ресурсов на В1/В3. Но: эту админку через 1-2 спринта всё равно нужно будет переделать (long-lived JWT + Basic Auth не масштабируется).

**Если выбирать B2 — обязательно сначала мигрировать FTP → rsync** (AUDIT P2-5), иначе clean-slate сожрёт серверный конфиг.

## Acceptance criteria (общие, для любого варианта)

- [ ] Ольга может через UI создать новый блокнот, вопрос, город.
- [ ] Не-админ не может (тест: открыть в инкогнито без логина, попробовать POST через DevTools).
- [ ] PIN-кодов 0000/1111 нет нигде в репо и в бандле (это уже выполнено в SEC-PINS).
- [ ] meetings-фронт публичной части не зависит от админки (read-only работает без логина).
- [ ] Smoke на стейджинге / dev-инстансе пройден.

---

## До решения по варианту

Ольга **временно не может** загружать тетради/вопросы/города. Это сознательная регрессия, согласованная стратегом в SEC-PINS Variant A. Старые данные продолжают читаться публикой.

Если Ольге **очень срочно** нужно что-то добавить (новая тетрадь к запуску курса, новый вопрос дня), есть **manual escape hatch**: попросить Garden-стратега / Claude Code в Garden выполнить INSERT через psql с owner-ролью. Это не масштабируется, но как разовая мера — работает.
