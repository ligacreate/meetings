## A. Все write-операции в meetings — текущий каталог

**Метод:** `grep -RIn "method: '(POST|PATCH|DELETE)'"` по `src/`. Дополнительно прошёл по `window.open`, `localStorage.setItem`, `fetch(...)` чтобы убедиться, что других write-каналов нет.

### A.1 Сводка
- **Все** PostgREST-write вызовы в meetings идут под admin-flow с PIN'ом. **Ни одного legitimate write от рядового посетителя.**
- Никаких клиентских регистраций на событие, лайков, комментариев, форм обратной связи в коде нет.
- Локальный side-effect: `localStorage.setItem` — кеш данных + `events_debug` флаг. Это не БД-write.
- Yandex.Metrika в [index.html:18-27](../index.html#L18-L27) — телеметрия через её скрипт, не наш fetch.

### A.3 Что можно резать целиком vs что должно стать legitimate JWT-write

Зависит от итога **ARCH_CHECK + 1-минутной верификации Ольги** (см. [ARCH_CHECK_meetings_admin_2026-05-05.md](ARCH_CHECK_meetings_admin_2026-05-05.md) §⚠️ critical contextual finding).

**Сейчас уверенно режем целиком (#1-5 из таблицы):**
- Все вызовы из EventsAdmin (5 шт). Дублируются Garden, к тому же POST events напрямую — архитектурный анти-паттерн (events — read-mirror таблицы `meetings`).

**Условно режем целиком (#6-13):**
- Если Ольга подтверждает «инструменты не работают» (ожидаю 401/42501) — режем все 13.
- Если работают — оставляем под legitimate JWT-write через Telegram-Auth, как Garden:
  - notebooks: #8, #9, #10
  - questions: #11, #12, #13
  - cities: #6, #7

**Никогда не было legitimate public-write в коде.** Регистрация на событие, оплата, лайки — нет. Это совпадает с ролью meetings как «публичной читалки».

## B. 17 npm vulns — классификация по риску breaking changes

Вытащил из `npm audit --json` (поле `fixAvailable.isSemVerMajor`).

### B.1 Headline

**Все 17 уязвимостей чинятся `npm audit fix` без `--force`.** Поле `isSemVerMajor` = `false` у каждой. Реально нужен только `npm audit fix` (минор/патч-апгрейды). **Major-bump не требуется ни для одной.**

Это **существенно мягче**, чем казалось из объёма списка. SEC-таск по vulns можно ставить в P2 (не P1), потому что:
- Все runtime-vuln'ы транзитивные через 3 direct deps (`postcss`, `react-router-dom`, `vite`) — все в свежих 6.x/8.x/5.x major-линиях, обновляются на patch.
- 0 рисков breaking-changes по semver.

### B.4 Реалистичный риск runtime-XSS из react-router 6.30.2

Заголовок «React Router XSS via Open Redirects» звучит страшно. Реальный профиль:
- Уязвимость через `useSearchParams` + редирект на пользовательский URL.
- В meetings нет роутов с редиректом на user-supplied URL: единственный `useNavigate('/')` в [App.tsx:17](../src/App.tsx#L17), без user input.
- → реальный риск low. Но фикс minor — всё равно делаем.

### B.5 Рекомендация для SEC-таска

**Severity: P2** (не P1, как было в исходном AUDIT — пересмотр после уточнения по semver).

Acceptance criteria:
- [ ] `npm audit` возвращает 0 high + 0 moderate.
- [ ] `npm run build` проходит зелёным.
- [ ] Smoke на стейджинге (или, поскольку стейджинга нет — на dev-инстансе перед FTP-deploy).

Команда:
```sh
npm audit fix       # без --force
npm run build
```

Время: 10-15 минут на верификацию + 1 деплой.

**Привязка к этапу 4:** делаем **после** этапа 4. Если перепутать — апгрейд `react-router-dom` 6.30.2 → 6.30.x может потенциально что-то сломать (хоть и patch), и придётся отделять регрессии этапа 4 от регрессий апгрейда. Сначала фича, потом гигиена.
